## Implementation Plan

### Step 1: Create a SQL DDL script in the FriedFish workspace

Write a single `ddl/create_tables.sql` file that creates all tables in `MET_SCRATCH.THAMES`:

- `STATIONS` тАФ VARCHAR PK, scalar columns, GEOGRAPHY point derived from lat/long
- `STATION_TYPES` тАФ composite PK (station_id, type_uri)
- `STATION_STATUSES` тАФ composite PK (station_id, status_uri)
- `MEASURES` тАФ VARCHAR PK, FK to stations
- `READINGS` тАФ composite PK (measure_id, date_time), FK to stations, clustered by (station_id, date_time)
- `RIVER_LINKS` тАФ VARCHAR PK, GEOGRAPHY column (LINESTRING from WKT/GeoJSON)
- `CATCHMENT_BOUNDARIES` тАФ VARCHAR PK, GEOGRAPHY column (parsed from GeoJSON), area_km2

### Step 2: Create views

- `RIVER_THAMES` тАФ simple WHERE filter on watercourse_name/watercourse_name_alternative
- `RIVER_THAMES_CONNECTED` тАФ recursive CTE (port directly from the Postgres version, Snowflake supports recursive CTEs)

### Step 3: Create a Python export script

Write `scripts/export_to_snowflake.py` that:
1. Connects to the local Postgres
2. Exports each table to CSV/Parquet (Parquet preferred for type fidelity and compression)
3. For geometry columns: exports as WKT strings (Snowflake's `TRY_TO_GEOGRAPHY(wkt)` or `ST_GEOGRAPHYFROMWKT()` handles conversion)
4. For `catchment_boundaries.geometry`: exports the GeoJSON dict as a string for `TRY_TO_GEOGRAPHY()`

### Step 4: Create a load script

Write `scripts/load_to_snowflake.sql` that:
1. PUTs the exported Parquet files to `@MET_SCRATCH.THAMES.GEO` (or a new `@MET_SCRATCH.THAMES.LOAD` stage)
2. COPY INTO each table from the staged files, with appropriate transforms for GEOGRAPHY columns

### Step 5: Upload DDL/load scripts to FriedFish workspace

PUT the SQL scripts into the workspace so they're accessible from Snowsight.

## Design decisions

- **GEOGRAPHY vs GEOMETRY:** Use Snowflake's GEOGRAPHY type (spherical, WGS84) тАФ matches the SRID 4326 in Postgres and is consistent with your existing `THAMES_GEOM` and `PSEUDO_SITES_GEO` tables
- **Parquet format:** Better than CSV for preserving types (timestamps, nulls, floats) and compresses well for the ~500k readings rows
- **Clustering key on READINGS:** `(STATION_ID, DATE_TIME)` for the most common analytical access pattern
- **No foreign keys enforced:** Snowflake FKs are informational only (not enforced), but we'll declare them for documentation/optimiser hints
